Zorvyn FinTech Backend Project Summary (Current State)
Date: 2026-04-01

1) Project Purpose
This project is a backend API for a finance dashboard system.
It provides:
- Authentication and role-based access control (VIEWER, ANALYST, ADMIN)
- User management (including status ACTIVE/INACTIVE)
- Financial record management with validation, filtering, pagination, and soft delete
- Dashboard analytics built with MongoDB aggregations
- Seed/demo data and integration tests

The codebase follows an MVC-style modular structure with a separate service layer for dashboard analytics.


2) Tech Stack
Runtime and API:
- Node.js (CommonJS)
- Express 5

Database:
- MongoDB
- Mongoose

Auth and Security:
- jsonwebtoken (JWT)
- bcrypt (password hashing)
- express-rate-limit (auth endpoint rate limiting)

Configuration and Dev:
- dotenv
- nodemon

Testing:
- jest
- supertest


3) NPM Scripts
From package.json:
- npm start -> node src/server.js
- npm run dev -> nodemon src/server.js
- npm test -> jest --runInBand
- npm run seed -> node src/scripts/seed.js


4) High-Level Request Flow
1. Request reaches Express app in src/app.js.
2. Route and middleware are applied (auth, role checks, validation, rate limit where needed).
3. Controller handles orchestration and response formatting.
4. For dashboard endpoints, controller delegates aggregation/business logic to src/services/dashboardService.js.
5. Mongoose reads/writes MongoDB.
6. Global error handlers return structured errors for unknown routes and failures.


5) Access Control Model (Current)
Role definitions in code comments and middleware:
- VIEWER: read-only dashboard access
- ANALYST: read records + dashboard
- ADMIN: full access

Effective access:
- Dashboard routes: VIEWER, ANALYST, ADMIN (GET-only)
- Records read (GET /records, GET /records/:id): ANALYST, ADMIN
- Records write (POST/PATCH/DELETE): ADMIN only
- User status management: ADMIN only


6) Folder Structure and File Responsibilities

Root Files
- package.json
  - project metadata, scripts, dependencies, jest config
- package-lock.json
  - lockfile for reproducible installs
- .env
  - local runtime env vars (not committed)
- .env.example
  - documented env template: PORT, MONGO_URI, JWT_SECRET, NODE_ENV
- .gitignore
  - excludes node_modules, .env, npm debug logs
- info.md
  - assignment/problem context
- summary.txt
  - this architecture and implementation summary

src/
- app.js
  - creates Express app
  - applies express.json middleware
  - mounts route modules
  - mounts dashboard routes at both /dashboard and /api/dashboard
  - applies notFound and errorHandler middleware

- server.js
  - loads dotenv
  - connects DB via connectDB
  - starts HTTP server
  - handles unhandled promise rejections
  - import-safe: does not auto-start when required by tests

src/config/
- db.js
  - connectDB function
  - reads MONGO_URI
  - connects Mongoose
  - logs host on success
  - logs and exits on connection failure

src/models/
- User.js
  - fields: name, email, password, role, status
  - email format validation + uniqueness
  - password select false by default
  - pre-save bcrypt hashing
  - comparePassword helper method

- FinancialRecord.js
  - fields: userId, amount, type, category, date, note, isDeleted
  - amount > 0 validation
  - type enum INCOME/EXPENSE
  - soft delete via isDeleted flag
  - indexes for filter/query optimization:
    - category/date/type
    - isDeleted + date
    - isDeleted + type + date
    - isDeleted + category + date

src/middleware/
- authMiddleware.js
  - protect: JWT bearer validation + user lookup + ACTIVE status check
  - authorizeRoles(...roles): role guard

- authRateLimiter.js
  - express-rate-limit config for auth endpoints
  - windowMs: 15 * 60 * 1000
  - max: 10 per IP
  - standardHeaders enabled
  - returns 429 with message:
    "Too many requests, please try again later"

- financialRecordValidation.js
  - validateFinancialRecord: strict validation for create payload
  - validateFinancialRecordUpdate: partial validation for patch payload

- errorHandler.js
  - notFound middleware
  - global error formatter (stack hidden in production)

src/controllers/
- authController.js
  - POST register logic
  - POST login logic
  - input validation and consistent auth responses

- userController.js
  - PATCH user status
  - validates allowed statuses and handles invalid user id

- financialRecordController.js
  - createFinancialRecord
  - getFinancialRecords with:
    - type/category/date-range filters
    - search across category + note (case-insensitive, regex escaped)
    - page/limit pagination
    - filtered totalCount
  - getFinancialRecordById
  - updateFinancialRecord (partial fields only)
  - deleteFinancialRecord (soft delete)
  - consistent error response helpers

- dashboardController.js
  - thin controllers for dashboard endpoints:
    - getSummary
    - getCategoryBreakdown
    - getRecentActivity
    - getMonthlyTrends
  - delegates to dashboardService
  - consistent dashboard error responses

- healthController.js
  - API health endpoint response with timestamp

src/services/
- dashboardService.js
  - all dashboard aggregation/business logic lives here
  - shared filter builder with validation:
    - type/category/startDate/endDate
    - invalid date format and invalid date range checks
  - getSummary:
    - totalIncome, totalExpense, netBalance
  - getCategoryBreakdown:
    - grouped by category
    - total, incomeTotal, expenseTotal
    - sorted descending by total
  - getRecentActivity:
    - latest records sorted by date desc
    - limit restricted to 5 or 10
    - minimal fields: amount, category, type, date
  - getMonthlyTrends:
    - grouped by year and month using $year and $month
    - outputs month label + income + expense timeline
  - query optimization using projection before grouping where relevant

src/routes/
- authRoutes.js
  - POST /auth/register (with authRateLimiter)
  - POST /auth/login (with authRateLimiter)

- userRoutes.js
  - PATCH /api/users/:id/status (ADMIN)

- recordRoutes.js
  - GET /records (ANALYST, ADMIN)
  - POST /records (ADMIN)
  - GET /records/:id (ANALYST, ADMIN)
  - PATCH /records/:id (ADMIN)
  - DELETE /records/:id (ADMIN)

- dashboardRoutes.js
  - router-level protect + authorizeRoles(VIEWER, ANALYST, ADMIN)
  - GET /dashboard/summary
  - GET /dashboard/category-breakdown
  - GET /dashboard/recent
  - GET /dashboard/trends
  - compatibility aliases:
    - /dashboard/recent-activity
    - /dashboard/monthly-trends

- healthRoutes.js
  - GET /api/health

src/scripts/
- seed.js
  - connects DB
  - clears users and financial records
  - inserts 3 demo users (ADMIN/ANALYST/VIEWER)
  - inserts 10 demo records assigned to ADMIN user
  - uses existing User model logic so passwords are hashed
  - logs summary and exits process

src/utils/
- generateToken.js
  - generates JWT containing user id and role
  - requires JWT_SECRET
  - token expiry: 1d

tests/
- auth.test.js
  - Jest + Supertest integration tests:
    - register success and no password in response
    - login success with token
    - record create success with admin token
  - uses separate test DB URI strategy (MONGO_URI_TEST if provided, else derived _test URI)
  - clears collections after each test

- Invalid ObjectId handled explicitly where needed
- Search input validation:
  - trim
  - ignore empty/whitespace
  - max length 50
  - regex escaped special characters
- Dashboard query validation:
  - date format checks
  - startDate <= endDate enforcement
