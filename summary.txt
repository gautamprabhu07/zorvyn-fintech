Zorvyn FinTech Backend Project Summary (Current State)
Date: 2026-04-01

1) Project Purpose
This is a Node.js + Express backend for a finance dashboard system with role-based access control.
It supports:
- User authentication and role management
- Financial record CRUD with filtering, pagination, and soft delete
- Dashboard analytics endpoints (summary, category breakdown, recent activity, monthly trends)

The backend is designed in a clean modular MVC style with a service layer for dashboard aggregations.


2) Tech Stack
Core:
- Node.js (CommonJS)
- Express 5
- MongoDB + Mongoose

Security and Auth:
- jsonwebtoken (JWT)
- bcrypt (password hashing)

Config and Dev:
- dotenv (environment variables)
- nodemon (development auto-restart)


3) High-Level Architecture
- Entry point: src/server.js
- App composition and middleware wiring: src/app.js
- Database connection: src/config/db.js
- Domain models: src/models/*
- Controllers: src/controllers/*
- Business/aggregation logic: src/services/dashboardService.js
- Routes: src/routes/*
- Shared middleware: src/middleware/*
- Utility helpers: src/utils/*

Flow overview:
1. Request enters Express app.
2. Route-level middleware runs (auth, role checks, validators).
3. Controller handles request/response orchestration.
4. For dashboard analytics, controller delegates aggregation logic to service layer.
5. Mongoose performs database operations.
6. Response is returned in structured JSON.


4) Folder Structure and Responsibilities
Root:
- package.json: scripts, dependencies, project metadata
- package-lock.json: dependency lockfile
- .env / .env.example: runtime configuration variables
- .gitignore: ignore secrets and dependencies
- info.md: assignment context and requirements snapshot
- summary.txt: this project summary

src/
- app.js: registers global middleware and mounts all route modules
- server.js: loads env, connects DB, starts HTTP server, handles unhandled promise rejections

src/config/
- db.js: Mongoose connection using MONGO_URI, success/failure logging, hard exit on failure

src/models/
- User.js:
  - Fields: name, email, password, role, status
  - Email validation and uniqueness
  - Password hash pre-save hook
  - comparePassword method
  - Password hidden by default (select: false)
- FinancialRecord.js:
  - Fields: userId, amount, type, category, date, note, isDeleted
  - Amount positive validation
  - Type enum INCOME/EXPENSE
  - Soft delete flag (isDeleted)
  - Indexes for date/type/category and dashboard filtering performance

src/middleware/
- authMiddleware.js:
  - protect: validates JWT from Authorization Bearer token
  - fetches user from DB and enforces ACTIVE status
  - attaches req.user = { id, role }
  - authorizeRoles(...roles): role-based access guard
- financialRecordValidation.js:
  - validateFinancialRecord: create payload validation
  - validateFinancialRecordUpdate: partial update validation
- errorHandler.js:
  - notFound handler for unknown routes
  - centralized error formatter

src/controllers/
- authController.js:
  - register: validates input, checks existing user, creates user
  - login: validates input, verifies password, checks ACTIVE status, returns JWT + safe user data
- userController.js:
  - updateUserStatus: ADMIN-level status management (ACTIVE/INACTIVE)
- financialRecordController.js:
  - createFinancialRecord
  - getFinancialRecords (filters + pagination + excludes deleted)
  - getFinancialRecordById
  - updateFinancialRecord (partial updates)
  - deleteFinancialRecord (soft delete)
  - uses consistent error responses and status codes
- dashboardController.js:
  - thin controllers for dashboard APIs
  - delegates all analytics logic to dashboardService
  - handles service/controller errors uniformly
- healthController.js:
  - basic API health response with timestamp

src/services/
- dashboardService.js:
  - getSummary: total income, total expense, net balance via aggregation
  - getCategoryBreakdown: grouped totals per category, includes income/expense split
  - getRecentActivity: latest records (limit 5 or 10), minimal fields only
  - getMonthlyTrends: monthly grouped income/expense timeline using $year/$month
  - shared query filter builder with validation for type/category/startDate/endDate

src/routes/
- authRoutes.js:
  - POST /auth/register
  - POST /auth/login
- userRoutes.js:
  - PATCH /api/users/:id/status (ADMIN)
- recordRoutes.js:
  - GET /records (ANALYST, ADMIN)
  - POST /records (ADMIN)
  - GET /records/:id (ANALYST, ADMIN)
  - PATCH /records/:id (ADMIN)
  - DELETE /records/:id (ADMIN, soft delete)
- dashboardRoutes.js:
  - GET /dashboard/summary (ANALYST, ADMIN)
  - GET /dashboard/category-breakdown (ANALYST, ADMIN)
  - GET /dashboard/recent (ANALYST, ADMIN)
  - GET /dashboard/trends (ANALYST, ADMIN)
  - compatibility aliases:
    - /dashboard/recent-activity
    - /dashboard/monthly-trends
  - also mounted under /api/dashboard in app.js for compatibility
- healthRoutes.js:
  - GET /api/health

src/utils/
- generateToken.js:
  - creates JWT payload with user id and role
  - uses JWT_SECRET and 1 day expiry


5) Security and Access Control
- Passwords are hashed with bcrypt before save.
- Password is not returned in API responses by schema select:false and response shaping.
- JWT-based authentication with Bearer token.
- Role-based middleware enforces endpoint permissions.
- Inactive users are blocked at auth middleware and login checks.

Role behavior currently implemented:
- VIEWER: blocked from dashboard analytics routes and admin operations
- ANALYST: can read records and dashboard analytics
- ADMIN: full access including create/update/delete records and user status management


6) Data and Query Features
Financial records:
- Validation on create/update
- Filtering by type, category, date range
- Pagination (page, limit)
- Soft delete (isDeleted=true)
- List and analytics exclude deleted records

Dashboard analytics:
- Date range filtering using startDate/endDate
- Query validation and clear error messages
- Aggregation pipelines with projection and grouping
- Monthly trends grouped using $year and $month


7) Environment Variables
Defined in .env.example:
- PORT
- MONGO_URI
- JWT_SECRET
- NODE_ENV


8) API Behavior and Error Handling
- Uses appropriate status codes such as 200, 201, 400, 401, 403, 404, 409, 500.
- Validation errors return meaningful messages.
- Invalid ObjectId and date-range errors are handled explicitly.
- Unknown routes are handled by notFound middleware.

- Clear modular backend structure
- Separation of concerns (routes/controllers/services/models/middleware)
- Secure auth + RBAC
- Financial CRUD + filtering + pagination + soft delete
- Dashboard aggregations implemented with MongoDB pipelines
- Good baseline production-like patterns for an internship assignment

- No dedicated automated test suite is present yet (manual smoke tests were used during development).
