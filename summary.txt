Zorvyn FinTech Full Stack Project Summary (Current State)
Date: 2026-04-02

1) Project Purpose
This project is a complete finance dashboard system with:
- A secured backend API (Node.js + Express + MongoDB)
- A role-aware frontend application (React + Vite + Tailwind)

The system supports:
- Authentication and JWT-based authorization
- Role-based access (VIEWER, ANALYST, ADMIN)
- Financial record management with validation, filtering, search, pagination, and soft delete
- Dashboard analytics (summary, category breakdown, recent activity, monthly trends)
- Frontend UX states (loading, error, retry, empty-state handling)
- Seed data and integration tests for core API behavior


2) Architecture Overview
Backend (API layer):
- Express app with MVC-style modular structure
- Thin controllers and service-layer analytics logic
- Mongoose models with indexes and validation
- Middleware-driven auth, role checks, validation, rate limiting, and error handling

Frontend (UI layer):
- React app with route-level protection
- Auth context for global auth and role state
- Centralized axios client with interceptors
- Centralized API modules per domain (auth, records, dashboard)
- Global app layout and reusable Tailwind utility classes

5) Backend: Detailed Implementation

5.1) Server/App Bootstrap
- src/server.js:
  - Loads dotenv
  - Connects MongoDB before listen
  - Handles unhandled promise rejections
  - Export-safe startup for test environments

- src/app.js:
  - JSON body parsing
  - Root welcome endpoint
  - Route mounting:
    - /api (health)
    - /auth
    - /api/users
    - /records
    - /dashboard and /api/dashboard (both supported)
  - notFound + global error handler middleware

5.2) Database and Models
- src/config/db.js:
  - ConnectDB with env validation and startup fail-fast behavior

- src/models/User.js:
  - Fields: name, email, password, role, status
  - Roles: VIEWER, ANALYST, ADMIN
  - Status: ACTIVE, INACTIVE
  - Email validation + unique index
  - Password hidden by default (select: false)
  - Pre-save bcrypt hashing
  - comparePassword method

- src/models/FinancialRecord.js:
  - Fields: userId, amount, type, category, date, note, isDeleted
  - amount > 0 validation
  - type enum INCOME/EXPENSE
  - soft delete via isDeleted flag
  - query indexes for list/filter/date use cases

5.3) Middleware
- src/middleware/authMiddleware.js:
  - protect:
    - bearer token validation
    - jwt verification
    - user existence check
    - ACTIVE status enforcement
  - authorizeRoles(...roles): role-based guard

- src/middleware/authRateLimiter.js:
  - Applied to auth register/login routes
  - 15-minute window, max 10 requests/IP

- src/middleware/financialRecordValidation.js:
  - create/update payload validation

- src/middleware/errorHandler.js:
  - notFound middleware
  - production-safe global error output

5.4) Controllers and Services
- src/controllers/authController.js:
  - register + login
  - input validation and normalized payload handling
  - login returns token and user data including role

- src/controllers/userController.js:
  - admin-only status updates for users

- src/controllers/financialRecordController.js:
  - create/read/update/delete (soft delete)
  - filters: type, category, startDate, endDate
  - search: category + note (regex escaped, case-insensitive)
  - pagination: page + limit + totalCount + totalPages

- src/controllers/dashboardController.js:
  - thin orchestration over service methods
  - summary/category/recent/trends endpoints

- src/services/dashboardService.js:
  - getSummary: totalIncome, totalExpense, netBalance
  - getCategoryBreakdown: totals + income/expense per category
  - getRecentActivity: latest records (limit 5/10)
  - getMonthlyTrends: month-level income/expense output
  - shared filter builder with date/type/category validation

5.5) Routes and Access Matrix
- src/routes/authRoutes.js:
  - POST /auth/register
  - POST /auth/login
  - rate limited

- src/routes/userRoutes.js:
  - PATCH /api/users/:id/status (ADMIN)

- src/routes/recordRoutes.js:
  - GET /records, GET /records/:id -> ANALYST, ADMIN
  - POST /records, PATCH /records/:id, DELETE /records/:id -> ADMIN

- src/routes/dashboardRoutes.js:
  - All endpoints protected and allowed for VIEWER, ANALYST, ADMIN
  - /summary
  - /category-breakdown
  - /recent (+ alias /recent-activity)
  - /trends (+ alias /monthly-trends)

- src/routes/healthRoutes.js:
  - GET /api/health

5.6) Seed and Tests
- src/scripts/seed.js:
  - Clears users + records
  - Inserts demo users (ADMIN, ANALYST, VIEWER)
  - Inserts 10 sample records

- tests/auth.test.js:
  - Register success
  - Login success (token issued)
  - Admin record creation with token

7) Security and Access Behavior (End-to-End)
Backend enforcement:
- Access rules are enforced server-side using protect + authorizeRoles.

Frontend enforcement:
- ProtectedRoute blocks unauthenticated access.
- allowedRoles on /records blocks VIEWER in the client route layer.
- axios interceptor auto-logout on 401 to prevent stale sessions.

Role behavior summary:
- VIEWER:
  - dashboard allowed
  - records route denied in frontend and backend
- ANALYST:
  - dashboard allowed
  - records allowed (read endpoints)
  - write endpoints denied by backend; write UI hidden/disabled
- ADMIN:
  - full dashboard + records access
  - write endpoints allowed by backend
